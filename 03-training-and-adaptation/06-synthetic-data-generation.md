## 在“数据墙”之后：合成转向

前沿模型（Llama 4、GPT-5.5、Claude Opus 4.7、Gemini 3.1 Pro）是在 100T+ tokens 上训练的。单靠人类文本，根本不足以支撑这种规模扩展。  
**现实是：** 现在前沿微调（以及 10% 的预训练）所用训练混合物中，超过 **50%** 已经是合成数据。

| 来源 | 人类数据 | 合成数据 |
|--------|------------|----------------|
| **体量** | 固定（有限） | 无限 |
| **质量** | 可变（有噪声） | 可控（已净化） |
| **成本** | 高（人工标注者） | 低（推理/GPU） |
| **偏差** | 互联网的镜像 | 可手工平衡 |

## Evol-Instruct 模式

Evol-Instruct 是一个递归过程：LLM 接收一个简单指令，并把它演化为一个更复杂的指令。

**演化方向：**
1. **广度**：增加任务数量。  
2. **深度**：加入约束、复杂因素或多步逻辑。  
3. **去噪**：清理措辞，去除“AI 腔”。

```python
# Simple Instruction: "Write a function to add two numbers."
# Evolved Instruction: "Write a thread-safe Python class that performs 
# matrix addition with error handling and unit tests, adhering to PEP8."
```

## 宪法式 AI 与 AI 反馈（RLAIF）

由 Anthropic 开发，并在整个行业中被广泛采用，RLAIF 使用一份“宪法”（一组规则）来指导模型评估并改进自己的数据。

**流程：**
1. **提出**：模型 A 生成一个回答。  
2. **批评**：模型 B（宪法裁判）根据指南识别缺陷。  
3. **修订**：模型 A 基于批评生成更好的版本。  
4. **训练**：最终的（Prompt，Revise）对被加入 SFT 集。

## 可验证的合成数据

合成数据最大的风险是 **模型坍塌**（Model Collapse，即模型学习到自己的错误）。  
**2025 解决方案**：聚焦那些无需 LLM 就能验证“真值”的领域。

- **数学**：使用形式化验证（Lean/Isabelle）或 Python 执行来验证答案。
- **代码**：让生成的代码在测试用例上运行（单元测试）。
- **RAG**：使用“黄金上下文”生成问题，使答案明确写在文本中。

## 去偏与多样性

合成数据用于“填补”人类数据中的空白。
- **语言**：通过翻译概念模板，为低资源语言（例如斯瓦希里语、马拉地语）生成高质量文本。
- **逻辑**：创建 1,000,000 个特定逻辑谬误的变体，以“加固”模型对它的抵抗能力。

## 面试题

### Q: 在使用合成数据训练时，“模型坍塌”有什么风险？

**强回答：**  
模型坍塌发生在模型使用其早期版本生成的数据进行训练时。因为模型的分布比真实世界更窄（它对某些词和模式存在偏好/偏差），训练循环就会变成错误和乏味内容的“正反馈循环”。通过 2025，我们可以通过以下方式缓解这一点：  
1. 混入 5-20%“金牌”人类认证数据。  
2. 使用“可验证”的奖励（数学/代码），让错误永远不会被学进去。  
3. 使用更强大的“教师”模型为“学生”模型生成数据。

### Q: 你如何确保一个 1000 万行合成数据集的*质量*？

**强回答：**  
我们使用一个 **多阶段过滤流水线**：  
1. **语义去重**：使用嵌入向量移除近乎相同的簇。  
2. **LLM 作为裁判**：抽样 1% 的数据，并让一个更强的模型（例如 GPT-5.2）从逻辑和安全性方面打分。  
3. **困惑度过滤**：使用小模型计算文本的困惑度。如果太高（胡言乱语）或太低（重复/过于简单），就丢弃。  
4. **可验证执行**：如果数据包含代码或数学内容，必须通过本地编译器/解释器检查。
