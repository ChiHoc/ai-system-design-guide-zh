# 基准测试与排行榜

公开基准测试是该领域讨论模型能力的方式：MMLU、SWE-bench、GPQA、Arena Elo。它们有助于定位，也会让决策变得危险。本章会教你每个主要基准测试衡量什么，哪些仍能区分前沿模型、哪些已经饱和，以及如何批判性地阅读基准测试声明，避免被一个数字误导。

核心思想：**基准测试的定义和已知缺陷是稳定的；它的分数是易过期的。** 排行榜数字每周都在变化，会被污染、偏 favorable 的评测框架抬高，也越来越多地受到伪造条目的污染。因此，本页首先说明每个基准测试*衡量什么*以及它*如何失效*，并把具体分数视为有日期、有来源、需要重新验证的快照。关于评估你自己的系统（真正预测生产质量的东西），请参阅 [LLM Evaluation（大语言模型评估）](01-llm-evaluation.md)。

## 如何阅读本页

下面每个基准测试都会列出它**衡量什么**、它的**格式**、它的**饱和状态**（前沿模型集群是否已经高到分数差异只是噪声？），以及它的**已知缺陷**。如果某个当前分数有助于定位，它会带着来源和“验证”标记给出，因为：

- **供应商自报分数高于独立排行榜。** 实验室会在它能找到的最佳评测框架、最佳努力设置和无上限基础设施下报告自己的模型。这个数字不能与另一家实验室的数字或独立运行结果直接比较。只比较由*同一个*评测框架产生的数字。
- **2026 年的网络充斥着伪造的排行榜页面**，会编造模型和分数。如果你看到某个不认识的模型名被归因了前沿编码分数，在主来源（实验室文章、基准测试自己的排行榜）确认之前，先假定它是 SEO 垃圾内容。
- **标题数字隐藏了评测框架。** “SWE-bench 上 88%”如果没有智能体脚手架、工具访问、努力等级和输出 token 上限，就无法解读。

因此：使用*衡量内容*和*缺陷*列来推理一个基准测试，并把任何单一百分比视为有日期的数据点，而不是事实。

## 能力地图

基准测试按它们探测的能力分组。在每个分组中，该领域会持续淘汰已经饱和的基准测试，并用更难的后继者替代，因此仍在起区分作用的基准每年都会变化。

### 通用知识与语言

大多**已经饱和**：前沿模型聚集在 88-90% 以上，因此差异多是噪声。把这些保留为历史基线，而不是前沿区分器。

| 基准测试 | 衡量内容 | 状态 | 备注 |
|-----------|----------|--------|-------|
| **MMLU** | 57 个学科的多选学术知识（约 15.9k 道题） | 已饱和（约 90%+ 集群） | 历史上引用最多的基准测试。约 6.5% 的题目存在标签错误（据 MMLU-Redux）；严重受到网络污染。已从模型卡中逐步移除。 |
| **MMLU-Pro** | 更难的 MMLU 后继者：10 个选项而不是 4 个，偏重推理，约 12k 道题 | 接近饱和 | 为修复 MMLU 饱和问题而构建；现在也触及同样的天花板（顶级集群约 88-90%）。因饱和已从 Artificial Analysis 指数中移除。 |
| **MMLU-Redux** | 对 MMLU 的错误修正再标注（约 3k 项） | 已饱和 | 存在是为了*量化* MMLU 的错误率（发现 6.49% 错误，某些子集最高达 57%），而不是给前沿模型排名。 |
| **HellaSwag** | 常识句子补全 | 完全饱和（95%+） | 人类基线约 95%；自 GPT-4 时代起已被解决。 |
| **ARC-Challenge** | 小学科学多选题 | 已饱和（约 96%+） | 实际上已经解决。 |
| **WinoGrande** | Winograd-schema（温诺格拉德模式）指代消解 / 常识 | 已饱和（约 90%+） | 仍有标注伪影。 |
| **BBH (BIG-Bench Hard)** | 23 个最难的 BIG-Bench 任务，多步推理 | 使用 CoT（思维链）后已饱和（>90%） | 已被 **BIG-Bench Extra Hard (BBEH)** 取代，BBEH 是因为 BBH 饱和而构建的。 |
| **GLUE / SuperGLUE** | 较旧的 NLU（自然语言理解）任务套件 | 已退役 | 模型在 2021 年初就在 SuperGLUE 上超过了人类基线。 |

### 前沿推理

这些基准测试仍能区分领域顶部模型，按剩余空间大致排序如下。

| 基准测试 | 衡量内容 | 状态 | 备注 |
|-----------|----------|--------|-------|
| **HLE (Humanity's Last Exam)** | 覆盖 100+ 学科的封闭式专家问题，抗检索（约 2,500 道题，约 10% 多模态） | 活跃，区分度大 | 首要的前沿知识基准测试。无工具分数大致横跨 40-53%，因此差距可读。它在 2025 年 1 月发布时，SOTA（当前最佳）低于 10%，这说明前沿移动得有多快。注意工具与无工具配置会抬高并混淆比较。 |
| **GPQA-Diamond** | 198 道博士级生物/物理/化学问题，Google-proof（抗谷歌搜索） | 正压缩到饱和 | 设计目标是让有网络访问能力的熟练非专家得分约 34%，博士专家约 65-70%。前沿集群现在约 92-94%，因此作为顶级区分器正在变弱，但仍能区分中端模型，也是很好的 sanity check（合理性检查）。 |
| **ARC-AGI-2** | 抽象网格谜题推理，关注效率（约 400 个任务）；抗记忆 | 活跃，强区分 | **请仔细阅读：**人类评审组通过率为 85%，85% 是大奖*门槛*。聚合器会报告前沿模型“达到 85%”，但那是在非 ARC-Prize 评测框架下、使用大量测试时计算的自报数字。ARC-Prize-Verified 的天花板低得多（高成本下约在 50 多分中段）。绝不要把门槛当作已达成分数引用。 |
| **FrontierMath** | 研究级数学，专家数学家需要数小时到数天的问题（约 350 道题，私有；Tier 4 最难） | 活跃但快速攀升 | 开放式问题，答案为可自动验证的闭式形式（防猜）。v1 中约 42% 的问题有错误（已在 2026 年 6 月 v2 修复）；它部分由 OpenAI 资助，因此单一实验室数字需要治理层面的提示。Epoch 自己的表述是低于 70% 处于可达范围，因此应怀疑远高于该水平的聚合器分数。 |
| **CritPt** | 由 50+ 名物理学家编写，覆盖 11 个领域的研究级物理推理 | 活跃，天花板很低 | 大多数前沿模型在完整挑战上得分为个位数到约 45%，因此区分清晰。较新（2025 年末），正在成为标准科学推理组件。 |

### 数学

经典数学基准测试已经被解决；仍活跃的区分器是最新竞赛年份和研究级集合。

| 基准测试 | 衡量内容 | 状态 | 备注 |
|-----------|----------|--------|-------|
| **GSM8K** | 小学应用题 | 已饱和（>95%） | GSM-Symbolic 显示，当数字/从句被扰动时分数会下降，说明高分部分来自记忆。 |
| **MATH / MATH-500** | 竞赛数学（AMC/AIME 水平） | 已饱和（顶级约 99%） | MATH-500 是一个 500 题子集；规模小意味着顶端方差很高。 |
| **AIME (2024/2025/2026)** | 奥赛短答题，整数答案，每年 30 道题 | 年份 N 一旦公开就会饱和 | 有价值是因为每年题目都是新的（低污染），直到发布为止。只有 30 项，因此单年分数方差很高；优先使用平均运行结果。2024/2025 已饱和。 |
| **HMMT, Putnam** | 更难的奥赛 / 本科证明竞赛 | Putnam 证明题未饱和 | 证明评分依赖 LLM-judge（大语言模型裁判）；只看最终答案的捷径会夸大真实证明能力。 |

### 编码

代码片段基准测试已经不适合排名；智能体式、仓库规模的基准测试才是生产信号。

| 基准测试 | 衡量内容 | 状态 | 备注 |
|-----------|----------|--------|-------|
| **HumanEval / HumanEval+ / MBPP / MBPP+** | 从 docstring（文档字符串）合成单函数，并用单元测试检查 | 已饱和（前沿约 90-97%） | 规模很小且自 2021 年起公开，因此受到污染。HumanEval+ 增加了约 80 倍测试，以捕捉脆弱代码；如果 + 版本分数下降，说明弱测试曾让有 bug 的代码通过。不要用它给前沿模型排名。 |
| **SWE-bench Verified** | 解决真实 GitHub issue，使隐藏测试通过（人工验证的 500 个 issue 子集） | 接近饱和，受污染 | 2024-2026 年典型编码数字。但 OpenAI 自己的评估团队指出，超过 60% 的任务测试有缺陷，并且解法可以从任务 ID 原样复现。可用作“高于约 80% 即为前沿”的分层过滤器，不适合精细排名。仅评测框架选择就会让结果摆动 10-20 个百分点。 |
| **SWE-bench Pro** | 更难、抗污染的问题解决（私有 + copyleft + 保留仓库） | 未饱和，正在成为主要信号 | 分数比 Verified 低 25-35 个百分点，区分度好得多。OpenAI 现在用 Pro 而不是 Verified 指向编码能力。仍要阅读评测框架：供应商报告数字可能比同一模型在标准化评测框架上高约 17 个百分点。 |
| **SWE-bench Multimodal** | 解决视觉/UI 软件问题（JS 前端，包含损坏 UI 的截图） | 未饱和 | 测试真实视觉 grounding（视觉落地/对齐）；纯文本 SWE-bench 系统表现吃力。 |
| **SWE-rebench / SWE-bench-Live** | 持续更新、去污染的问题解决；任务晚于模型截止日期 | 未饱和 | 最强的污染审计。SWE-rebench 专门捕捉 Verified 上虚高但在新任务上崩塌的分数。 |
| **LiveCodeBench** | 按发布日期标注的竞争性编程题；只在截止日期之后的问题上计分 | 在新鲜窗口中未饱和 | 通过构造实现抗污染。竞争性编程不是软件工程，因此它预测算法推理，而不是仓库工作。 |
| **Aider Polyglot** | 跨 6 种语言的真实多文件编辑，必须输出有效编辑格式，2 次尝试 | 接近饱和（约 88%） | 衡量编辑格式可靠性，而不是仓库规模推理；Exercism 来源公开（污染）。 |
| **SciCode** | 分解为子问题的研究级科学编码 | 远未饱和 | 最难的主流编码基准测试。注意存在两种不兼容的评分约定（主问题 vs 子问题）；始终注明使用哪一种。 |
| **Terminal-Bench (2.x)** | 真实终端沙箱中的端到端智能体任务（构建、调试、系统管理） | 活跃，未饱和 | 分数与评测框架绑定（智能体 CLI 与模型同样重要）。 |

**为什么智能体式基准测试取代了 HumanEval：** HumanEval/MBPP 对每个前沿模型都在 90%+ 饱和，规模小且被记住了，并且单函数 docstring 到代码的任务不能预测真实工程能力。证据是：SWE-bench 发布时，同一代在 HumanEval 上取得高分的模型，解决率低于 2%。阅读仓库、运行测试、阅读堆栈跟踪并迭代补丁是另一种技能，因此该领域转向了 issue 解决和终端基准测试。

### 智能体与工具使用

这是变化最快的领域，因为智能体正是 2026 年生产价值所在。大多数这些基准测试尚未饱和。

| 基准测试 | 衡量内容 | 状态 | 备注 |
|-----------|----------|--------|-------|
| **BFCL (Berkeley Function Calling Leaderboard)** | 工具/函数调用准确率；V4 增加带网络搜索和记忆的智能体任务 | V1/V2 已饱和，V3/V4 未饱和 | 基于 AST（抽象语法树）的评分可能漏掉语义错误。注意过期的榜单镜像；使用官方 Berkeley 排行榜。 |
| **tau-bench / tau2-bench (Sierra)** | 智能体在领域策略（零售、航空、电信）下使用工具并与模拟用户对话 | 可靠性上未饱和 | 报告 **pass^k**（同一任务重新运行 k 次），这暴露了残酷的可靠性断崖：一个约 60% pass@1 的智能体可能跌到约 25% pass^8。这是最具生产相关性的工具使用信号，因为它衡量一致性，而不只是最佳情况。原始 tau2 有带 bug 的任务；使用修正后的分支。 |
| **GAIA** | 需要多步工具使用、网页浏览、文件处理的通用助手任务 | 顶端已被编排式集成系统饱和（约 92%） | 顶级条目是多模型集成和脚手架，而不是基础模型，因此它更多衡量编排而不是模型。 |
| **OSWorld-Verified** | 在真实操作系统上的计算机使用智能体，按执行结果评分 | 未饱和，接近人类区间 | OSWorld 的清理和标准化后继者（OSWorld 发布时最佳分数约 12%，人类约 72%）。供应商数字高于公开评测框架数字。 |
| **Online-Mind2Web / WebArena** | 实时网页智能体任务 | 混合 | Online-Mind2Web 的“进步幻觉”发现：一旦用透明方式评判，许多商业智能体表现不如一个 2024 年的学术基线。裁判方法差异极大，因此分数经常不可比。 |
| **GDPval** | 覆盖 44 个职业的真实经济价值知识工作，由人类专家评分 | 活跃，未饱和 | 面向未来的“它能否完成一天真实工作”信号；前沿模型以约低 100 倍的成本接近专家交付物质量。成对 Elo 变体（GDPval-AA）不同于胜率版本。 |
| **METR time-horizon** | 模型以 50% 可靠性完成的任务长度（以人类分钟计） | 活跃研究标准 | 不是排行榜，而是一种趋势：总体上时间范围约每 7 个月翻倍，在编码上更快。这是谈论智能体自主性增长最清晰的方式。 |

### 长上下文

核心发现：**标称上下文窗口夸大了可用上下文。** 应该先讲这一点，而不是规格表上的窗口大小。

| 基准测试 | 衡量内容 | 状态 | 备注 |
|-----------|----------|--------|-------|
| **NIAH（needle-in-a-haystack，大海捞针）** | 在不同深度和长度下检索单一事实 | 已饱和 / 琐碎 | 前沿模型得分约为 100%，这会带来虚假的信心。它是健全性检查，而不是区分器。 |
| **RULER（NVIDIA）** | 在受控长度下的检索、多跳追踪、聚合、QA；报告“有效长度” | 未饱和 | 有效长度与标称长度差距的主要来源：许多声称支持 128K 的模型，只能把质量保持到约 32-64K（标称值的 25-50%）。它是合成测试，所以要搭配基于散文的测试。 |
| **Fiction.LiveBench** | 深层叙事理解（心智理论、时间顺序、隐含推理），最高约 192K 个 token | 未饱和 | 最严苛的实用长上下文测试；大多数模型到 192K 时会跌到 80% 以下。规模很小（36 道题），因此噪声较大。 |
| **MRCR（multi-round coreference，多轮共指）** | 在多个近乎相同的大海捞针目标中进行区分，并返回第 i 个 | 未饱和，尤其是 8-needle | 分数会随着 needle 数量和长度陡降。 |
| **LongBench v2 / LongBench Pro** | 真实的长上下文理解与推理，8K-2M 个 token | 未饱和 | LongBench Pro 的发现：长上下文*优化*胜过原始参数规模扩展，并且每个模型的有效上下文都低于标称上下文。 |

设计文档里可以使用的一句话：*标称上下文窗口通常会夸大可用上下文；在 RULER 上，许多声称支持 128K 的模型只能把质量维持到约 32-64K，尽管前沿水平正在快速提升。* “context rot（上下文腐化）”研究（退化会在窗口上限之前很早出现，有时甚至出现反直觉结果：打乱的大海捞针集合比连贯文档表现更好）进一步支持按更小的有效预算来设计。参见 [Context Engineering（上下文工程）](../05-prompting-and-context/05-context-engineering.md)。

### 多模态

图像多选基准测试正在饱和；视频推理才是真正尚未解决的前沿。

| 基准测试 | 衡量内容 | 状态 | 备注 |
|-----------|----------|--------|-------|
| **MMMU / MMMU-Pro** | 跨 6 个学科的大学水平多模态推理 | 接近饱和 | MMMU-Pro 提高了难度（10 个选项、问题位于截图中的纯视觉题目），但存在严重的跨 harness（评测框架）分歧（同一模型在不同 harness 上被报告为 81% 和 94%），因此绝不要跨来源比较。 |
| **MathVista、DocVQA、ChartQA、MMBench** | 视觉数学、文档 QA、图表 QA、广义多模态 | 前沿模型基本饱和 | DocVQA/ChartQA 接近解决；宽松准确率评分会掩盖数值错误。 |
| **Video-MME / Video-MME-v2** | 视频理解 | v1 接近饱和，v2 远未饱和 | Video-MME-v2（2026）使用非线性分组评分，并显示出巨大的模型与人类差距，因此它是当前有效的多模态区分器。 |

### 事实性与指令遵循

| 基准测试 | 衡量内容 | 状态 | 备注 |
|-----------|----------|--------|-------|
| **IFEval** | 可程序化验证的指令遵循（“至少 400 个词”、有效 JSON、不能有逗号） | 基本饱和（约 90%+） | 只检查*可检查*约束，是指令遵循中很窄的一部分；容易被刷分。 |
| **SimpleQA / SimpleQA Verified** | 简短闭卷事实回忆，对抗性高难度；奖励校准后的拒答 | 未饱和 | 最高大约在 50 多的 F1 中段，因此幻觉远未解决。核心教训：能力更强不代表事实性更高。 |
| **TruthfulQA** | 抵抗常见误解的能力 | 老化 / 部分饱和 | 静态且知名，因此被污染；“真相”标签存在争议；可通过模棱两可的回答刷分。 |
| **FACTS Grounding** | 长篇回答是否完全由所给来源支持（无无依据断言） | 未饱和（最高约 0.88） | 由一组与参赛模型有共同谱系的 LLM（大语言模型）评判；提交项为自报。它是 RAG（检索增强生成）忠实性的正确基准家族。 |

|-----------|----------|--------|-------|
| **LMArena（Chatbot Arena Elo）** | 众包盲测成对偏好，以 Elo 报告 | 没有上限，但前 ~15 名压缩在约 25 个 Elo 点内 | 著名的偏好信号，也是最常被误用的信号。参见 [排行榜幻觉](#排行榜幻觉)。务必阅读**风格控制** Elo（回归剔除长度和格式偏差）及置信区间；~15-20 Elo 内的差异都是噪声。 |

## 批判性阅读基准测试

这才是真正重要的部分。任何人都能读排行榜；*正确*地读才是技能。

### 饱和

当前沿模型聚集在非常接近上限的位置，以至于分数差异落入噪声范围时，基准测试就饱和了。MMLU 是典型案例：GPT-4 在 2023 年初达到约 86%，此后前沿模型一直停留在 86-93%，因此 2 分的“胜利”往往只是提示词伪影（MMLU 分数会因提示措辞不同而变化 4-5%）。可用的工作信号是：**当领先者集中在约 3 分以内时，排名顺序是统计噪声，而不是能力差异。**

当一个基准测试饱和后，领域会用以下方式回应：(1) 更难的后继者（MMLU 之后是 MMLU-Pro，再之后是 HLE），(2) 私有或留出测试集，(3) 有时间门控的“live（实时）”基准测试，以及 (4) 复合指数。静态公开基准测试的有用寿命中位数不到约 2 年。

### 污染

基准测试是公开的，会被抓取进预训练数据，因此模型可能通过记忆而不是能力获得高分。证据很直接：重新推导 HumanEval 风格问题（EvoEval）使 51 个模型的分数下降约 39%；在 LiveCodeBench 上，某模型在截止日期前问题上的通过率约为 60%，在截止日期后问题上降到约 0%；OpenAI 发现 SWE-bench Verified 的解法可以从任务 ID 逐字复现。在多个多选 QA 基准测试中，测得的污染范围从 1% 到 45%，并且更大的模型从中获益更多。

抗污染设计包括：**time-gating（时间门控）**（只对模型截止日期之后发布的问题计分，如 LiveCodeBench 和 SWE-rebench）、**private held-out sets（私有留出集）**（FrontierMath、ARC-AGI-2；代价是不可复现），以及 **canary strings（金丝雀字符串）**（在数据集中植入唯一 token，如果模型复述它就标记污染）。检测方法（n-gram 重叠、成员推断、TS-Guessing 测验）都有失效模式；尤其是成员推断攻击，在真实预训练模型上几乎只比随机略好。实际做法是：对任何静态公开基准测试，都假设存在一定污染，并折扣其绝对数值。

### Harness 与脚手架方差

同一组模型权重会因提示词、是否可用工具、推理努力级别以及 agent（智能体）脚手架不同而产生 10-20 分的差异。Anthropic 测得，仅基础设施配置本身（RAM、并发，甚至一天中不同时间的 API 延迟）就会让 Terminal-Bench 结果移动约 6 分。这就是为什么**供应商自报分数高于独立排行榜**：实验室会报告它们为自家模型找到的最佳 harness 和努力级别，并运行在不受限制的基础设施上。由此得到的硬规则是：**绝不要把一个供应商的数字与另一个供应商的数字比较，也不要与独立排行榜比较。** 只有同一 harness 下的数字才可比较。而且推理努力并非单调关系，在一项大型 agent 研究中，更多思考在 36 个设置中的 21 个反而降低了准确率，因此供应商的“高努力”数字甚至不能与同一模型在独立默认努力运行下的结果比较。

### 排行榜幻觉

对 LMArena 的核心批评（Cohere 等，对约 2M 场对战、243 个模型的审计）发现了四个问题：供应商私下测试许多变体，只发布最好的（Meta 在 Llama-4 前测试了 27 个变体），这违反了 Elo 数学背后的无偏抽样假设；闭源供应商获得的对战数据远多于开源模型；你可以针对 Arena 分布训练，从而获得大量胜率增益；悄悄弃用的模型会扭曲排名。LMArena 的回应质疑了影响幅度（他们估计私下测试带来的提升约为 11 Elo，并会随着新投票累积而衰减），并指出过拟合数字是在静态代理上测得的，而不是实时人类榜。应把这呈现为**有争议但有证据支持。**

无论如何，实践建议都是一样的：把 Arena Elo 视为**通用聊天偏好**的度量，而不是正确性、事实性或困难推理的度量；始终使用风格控制榜（Arena 会奖励更长、更漂亮的回答）；阅读置信区间（前 ~15 名在统计上接近并列）；并把它作为三类信号之一，而不要单独使用。

### 基准测试到生产的差距

公开分数只有在三个条件同时成立时，才会预测你的生产表现：基准测试考察的任务与你的任务相似，测试集没有污染，并且基准测试尚未饱和。实践中这三点很少同时成立。对基准测试分数进行的主成分分析发现，单一“通用能力”因子只能解释约 50% 的方差；剩余部分是模型家族特性和噪声，这就是为什么两个通用能力相当的模型在*你的*任务上可能表现差异很大。高 GPQA 并不能保证在你的领域表现良好。

对于编码和 agent，最好的公开代理指标是 GPQA-Diamond 和 SWE-bench Verified（Aider Polyglot 和 AIME 风格集合也能干净地加载到通用能力上），但前提是 harness 匹配。每个实践者都会得出这个结论：**对你的决策来说，忽略排行榜，在你的数据上构建 eval（评测）。** 构建一个按功能、场景和 persona（用户画像）划分的黄金集；使用经过领域专家校准的二元 LLM-as-judge（LLM 作为评委）（用 precision（精确率）和 recall（召回率）衡量，而不是原始一致率）；并把能力与成本一起定价，因为没有任何公开基准测试包含成本信号。参见 [LLM Evaluation（LLM 评测）](01-llm-evaluation.md) 以及 [Whiteboard Exercises（白板练习）](../00-interview-prep/04-whiteboard-exercises.md) 中的 eval-pipeline 白板练习。

### 复合指数

由于任何单一基准测试都会在一两年内饱和，领域会用加权复合指标对前沿模型排名；当组件达到满分时，它们仍能保持区分力，并能抵抗对单个测试的过拟合：

- **Artificial Analysis Intelligence Index（Artificial Analysis 智能指数）** 对约 9 项评测加权，覆盖 agent、编码、科学推理和通用知识，并在组件饱和时重新版本化（它已经随时间移除了 MMLU-Pro、LiveCodeBench 和 AIME 2025）。
- **Epoch Capability Index（ECI，Epoch 能力指数）** 在约 1,000+ 项评测上拟合 item-response-theory（项目反应理论）模型，从统计上推断每个基准测试的难度，因此模型在*更难*的基准测试上表现好会获得更高分。
- **HAL（Holistic Agent Leaderboard，普林斯顿整体 Agent 排行榜）** 是面向 agent 的、具备成本意识的复合指标：它同时评分准确率*和*美元成本，在各模型上运行固定 harness，并用日志分析发现 agent 走捷径（从 arXiv 抓答案而不是解题）和 eval bug。它存在的原因是，以 10 倍成本换来 1% 的准确率提升并不算胜利。

复合指标适合回答“哪个模型总体最好”，但它们仍会继承组件的缺陷，因此要看清它们聚合了什么。

## 实用清单

当你阅读任何基准测试声明时：

1. **阅读 harness，而不只是数字。** 要求提供脚手架、工具访问、努力级别和输出 token 上限。裸百分比无法解释。
2. **检查置信区间。** 忽略噪声带内的差异。SWE-bench Verified 只有 500 道题（每题约 0.2%）；Arena 中低于约 15-20 Elo 的差距都是噪声。在配置匹配前，不信任任何低于约 3 分的差距。
3. **优先选择时间门控、私有或留出测试集。** 对静态公开测试集（MMLU、HumanEval、GSM8K），假设存在污染，并折扣绝对数值。
4. **绝不要跨 harness 或自报分数比较。** 只有同一 harness 下的数字才可比较。
5. **核对模型名称。** 如果某个前沿分数被归因于你找不到一手来源的模型，它很可能是编造的。已确认的 2026 月前沿家族：Claude（Fable 5、Opus 4.8/4.7、Sonnet 4.6、Haiku 4.5）、GPT-5.5、Gemini 3.1 Pro、DeepSeek V4、Llama 4、Kimi K2.6、Qwen 3.6、Mistral Medium 3.5、Grok 4.3。
6. **在信任排名前，三角验证三类信号**：静态学术 eval、人类偏好 arena，以及 agentic suite（智能体套件）。
7. **对你的决策，在你的数据上构建 eval。** 公开分数很少能预测你的领域。排行榜告诉你该把谁列入候选；你的黄金集告诉你该把谁上线。

## 2026 中哪些基准测试重要

用于判断 **Frontier General Capability（前沿通用能力）**：HLE、GPQA-Diamond（但要注意它的区分度正在变弱）、CritPt，以及复合指数（Artificial Analysis Intelligence Index、Epoch ECI）。忽略 MMLU、MMLU-Pro、HellaSwag、ARC-Challenge。

用于 **Coding（编码）和 Agents（智能体）**：SWE-bench Verified 可作为分层筛选器；SWE-bench Pro 和抗污染的实时变体（SWE-rebench、SWE-bench-Live）用于真实排序；Terminal-Bench 和 tau2-bench 用于评估工具使用与可靠性（关注 pass^k）；HAL 用于成本感知的智能体对比。忽略 HumanEval/MBPP。

用于 **Long Context（长上下文）**：优先看 RULER 和 Fiction.LiveBench，而不是 NIAH；设计时关注有效上下文，而不是宣传中的上下文长度。

用于 **Factuality（事实性）**：SimpleQA 用于闭卷幻觉评估，FACTS Grounding 用于 RAG（检索增强生成）忠实度；记住，更强的能力并不意味着更高的事实性。

用于 **Preference（偏好）**：采用带置信区间的风格控制 Arena Elo，或将 Arena-Hard-Auto v2 作为可复现代理指标。

至于 **你的产品**：以上都不够。构建你自己的黄金集和评审器。这个方向的前沿研究正在快速发展（智能体可靠性科学、声明级忠实度评估、模型能检测自己正在被测试的评测感知能力）；见 [研究雷达](../RESEARCH-RADAR.md)。

## 面试题

### 问：某供应商说他们的模型在 SWE-bench Verified 上得分 90%。在相信它能预测你的编码智能体质量之前，你会问哪些问题？

**优秀回答：**
首先看测试框架：使用了哪个智能体脚手架，哪些工具，什么努力级别，输出 token 上限是多少？相同权重仅因脚手架不同就会波动 10-20 分，而且供应商数字通常来自他们找到的最佳框架，因此这个 90% 不能和其他模型发布的数字直接比较。其次看污染：SWE-bench Verified 存在部分污染（OpenAI 发现可以从任务 ID 复现解法），所以我会要求看抗污染变体（SWE-bench Pro、SWE-rebench），这些基准上的分数通常低 25-35 分，并且区分度更好。第三看置信区间：它只有 500 个问题，所以几个百分点可能只是噪声。第四也是最重要的，看生产差距：即使 SWE-bench 数字是干净的，也只有在我的任务类似 GitHub issue 解决时，才可能预测我的代码库表现。我会用公开数字做入围筛选，最终用来自我们仓库的真实工单构建自己的黄金集，由校准过的评审器打分，并结合成本做决策。

### 问：为什么 MMLU 和 HumanEval 这类基准测试已经不再适合用于前沿模型排名？它们被什么取代了？

**优秀回答：**
它们已经饱和：前沿模型在 MMLU 上集中在 88-90% 以上，在 HumanEval 上达到 90%+，因此差异落在提示词措辞噪声范围内。它们也很小且公开，所以容易污染；重新推导 HumanEval 问题会让分数下降约 39%。而且它们衡量的构造过于简单：单函数的 docstring-to-code（文档字符串到代码）任务无法预测真实工程能力，这就是为什么同样在 HumanEval 上表现优异的模型，在 SWE-bench 刚推出时得分低于 2%。该领域用更难的后继基准（从 MMLU 到 MMLU-Pro 再到 HLE）、抗污染的时间门控基准（LiveCodeBench、SWE-rebench）、智能体和仓库规模基准（SWE-bench、Terminal-Bench、tau2-bench），以及在组件饱和后仍能保持区分度的复合指数取代了它们。

### 问：在为聊天产品选择模型时，你会如何负责任地使用 LMArena Elo？

**优秀回答：**
把它作为通用聊天偏好的一个信号，而绝不把它当作正确性或推理能力的度量。我会阅读风格控制榜单，因为原始 Arena 会奖励更长、格式更好的答案，不管它是否正确；我也会阅读置信区间，因为前十几名模型在统计上大致接近，差距约在 15-20 Elo 以内。我还会对排行榜幻觉效应打折：提供商会私下测试许多变体并发布最好的那个，所以一个刚登顶的新条目可能部分来自 best-of-N 运气。然后我会结合客观基准和智能体套件进行交叉验证，并最终在我自己的偏好数据上验证，因为 Arena 的提示词不是我的用户提示词。
