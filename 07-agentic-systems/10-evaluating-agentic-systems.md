# 评估 Agentic 系统

评估智能体在本质上与评估 RAG 不同。RAG 关注的是“准确性”，而智能体关注的是**“可靠性”、“效率”和“安全性”**。生产环境下的智能体评估依赖 **Trajectory Benchmarks（轨迹基准）** 和 **LLM-as-Judge（大模型作为评审者）** 来处理多步推理，并借助 Langfuse、LangWatch、Braintrust 和 Arize Phoenix 等工具，在原生追踪级别进行评分。

> [!NOTE]
> 对于标准的 RAG 评估（检索与生成指标），请参见 [06-retrieval-systems/09-advanced-retrieval-patterns.md](../06-retrieval-systems/09-advanced-retrieval-patterns.md) 和第 14 节。本章专注于智能体的*执行路径*。

## 评估转变

| 指标 | RAG 应用 | Agentic 应用 |
|--------|---------|-------------|
| **评估单位** | 单次响应 | **轨迹**（全部步骤） |
| **成功标准**| 基于语料一致性/忠实性 | 任务完成 / 逻辑健全性 |
| **复杂度** | 低（文本相似度） | 高（工具状态验证） |

---

## 轨迹基准

现代评估关注的是**“通往结果的路径”**。
1. **最优路径**：解决任务所需的最短工具序列。
2. **智能体路径**：实际采取的步骤。
3. **评分**：`Efficiency = (Optimal Steps / Agent Steps)`。得分为 `0.2` 表明智能体走了很多弯路或发生了过度循环。

**常见基准**：
- **SWE-bench**：修复 GitHub 问题（代码智能体）。
- **WebArena**：在菜单和表单中导航（浏览器智能体）。
- **GAIA**：通用工具使用任务（助手智能体）。

## 关键指标

### 1. 任务成功率（TSR）
最终状态正确的任务所占比例。
> [!IMPORTANT]
> 在资深级生产环境中，通过“错误路径”得到的“正确答案”得分为 0。

### 2. 动作成功率（ASR）
返回有效数据（不是错误或幻觉）的单个工具调用所占比例。

### 3. 每任务单位成本
每个完成目标的总代币 + 基础设施成本（沙箱、API 调用）。

## 用于步骤质量评估的 LLM-as-Judge

我们使用更强的模型（Claude Opus 4.7、GPT-5.5 reasoning）来复核较小智能体的**推理日志**。
- **思考质量**：智能体使用工具 X 的逻辑是否源于观察 Y？
- **冗余检查**：智能体是否重复执行了刚刚做过的搜索？
- **反馈闭环**：该“评审者”输出随后用于 **DPO（直接偏好优化，Direct Preference Optimization）**，以对齐智能体未来的行为。

## 生产环境评估

生产团队使用**影子执行**。
1. **V1 智能体** 响应用户。
2. **V2（实验）智能体** 在“隐藏沙箱”中执行同一查询。
3. **比较**：我们比较两条轨迹。如果 V2 能持续以更少步骤解决任务且没有安全违规，就将其提升到生产环境。

## 面试题

### 问：当环境是非确定性的（例如网页）时，如何评估一个智能体？

**高质量回答：**
我们使用**模拟环境**或**快照状态**。在高保真测试中，我们使用每次测试运行都会重置到干净状态的容器化浏览器。然后将智能体的轨迹与一个**参考轨迹**进行比较。如果环境是真实在线的，我们使用**基于状态的验证**——不比较文本，而是检查外部世界的状态（例如“数据库中是否有一条值正确的新记录”）。

### 问：为什么“走偏”（步骤过多）在高级别智能体设计中是致命失败？

**高质量回答：**
走偏会导致三类失败：1）**成本**：每一步都是一次 LLM 调用；2）**延迟**：每一步增加 2-5 秒；3）**熵**：轨迹越长，智能体遇到奇怪边界情况并触发幻觉的概率越高。标准修复方法是**步骤预算**：如果智能体在 10 步内没有解决任务，我们就终止它并升级给人工，以防止“Token 泄漏”。
