# 提示工程基础

提示工程（Prompt Engineering）是设计输入以引导大语言模型（LLM）行为的过程。它已经从“试错”演变为一种有纪律的架构实践，像 DSPy 这样的框架将其视为编译问题，而不是写作练习。

## 核心理念：意图 + 约束

有效的提示在于最大化**意图披露**，同时最小化**输出方差**。

1. **意图**：模型应当精确执行什么。
2. **约束**：模型应当明确避免什么（安全、语气、格式）。

**原则**：“提示工程就是自然语言编程。”把你的提示当作代码来对待（版本控制、单元测试）。

## 指令层级

生产系统使用分层消息结构：

| 角色 | 职责 | 细微差别 |
|------|----------------|--------|
| **系统** | 高层规则、人格设定、安全。 | 对前沿模型最具黏性（H-rank）。 |
| **开发者** | 技术性覆盖（例如格式）。 | 面向“不带偏见”模型的新角色。 |
| **用户** | 具体、动态的查询。 | 容易受到注入攻击；必须隔离。 |
| **助手**| 先前轮次的历史。 | “近期偏差”的来源。 |

## 角色提示

赋予人格不再只是“你是一名教师”。它是一种**能力锚点**（Capabilities Anchor）。

- **弱**：“你是一个程序员。”
- **强**：“你是一家 Tier-1 技术公司的高级软件工程师，专注于高并发 Rust 系统。你优先考虑内存安全和零成本抽象。”

**有效原因**：它将模型的注意力聚焦到与该高级专业能力相关的训练数据子集上，减少无关幻觉。

## 指令清晰性与分隔符

当前的前沿模型会处理海量上下文。分隔符帮助模型区分指令与数据。

```markdown
# Instructions
Analyze the following text for PII.

# Data to Analyze
--- START OF USER DATA ---
$USER_INPUT_HERE
--- END OF USER DATA ---

# Output Schema
{ "pii_found": boolean, "types": [] }
```

**可使用的分隔符**：XML 标签（`<context>`、`</context>`）、Markdown 标题（`#`）或三引号（`"""`）。

## 零样本与少样本效率

| 方面 | 零样本 | 少样本 |
|--------|-----------|----------|
| **延迟** | 最低（短提示） | 更高（示例令牌） |
| **准确性**| 可变 | 高（格式稳定） |
| **使用场景**| 简单聊天、总结 | 特定格式、细微逻辑 |

**策略**：如果模型是“前沿推理”模型（Claude Opus 4.7、带扩展思考的 GPT-5.5、DeepSeek-R2），使用**零样本 + 清晰的思维链（Chain-of-Thought，CoT）**。如果是小模型（8B），使用**少样本**来为其提供锚定。

## 面试题

### 问：为什么在现代大语言模型中，系统提示比用户提示权重更高？

**强回答：**
系统提示通常会因模型的架构训练（RLHF，基于人类反馈的强化学习）而被优先处理，并且在某些架构中可能被注入到一个特殊的“仅指令”嵌入空间中。从设计角度看，系统提示定义了交互的“宪法”。如果用户提示与系统提示相冲突（例如，要求提供炸弹配方），一个对齐良好的模型会被训练为优先遵循系统的“安全约束”，而不是用户的“任务意图”。

### 问：“Step-by-Step”提示优化是什么？

**强回答：**
在 2022 中，“一步一步思考”曾是一种触发思维链（CoT）的魔法短语。现代方法是**程序化思维链**（Programmatic CoT）。我们不再使用模糊短语，而是提供明确的推理里程碑：“1. 识别核心问题。 2. 列出约束条件。 3. 提出 3 个解决方案。 4. 选择最佳方案并说明理由。”这为模型的内部注意力提供了一条“确定性路径”，从而为生产级智能体带来更可靠的输出。
