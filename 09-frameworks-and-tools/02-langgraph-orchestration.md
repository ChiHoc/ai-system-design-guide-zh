# LangGraph 编排

LangGraph 是构建有状态、多智能体系统的**事实标准**。它在 2025 年末达到 v1.0，并在 2026 年初凭借其基于图的运行时在 GitHub stars 上超过了 CrewAI。不同于简单链式流程，LangGraph 支持**循环**、**状态持久化**以及**Human-in-the-Loop（人类参与闭环）**干预。

## 图的理念

在 2023 年，智能体还是“黑盒”。
如今，智能体是**图**。
一个图由以下部分组成：
- **节点**：Python 函数（LLM、工具或数据处理）。
- **边**：节点之间的路径。
- **条件边**：根据**状态**决定路径的逻辑。

---

## 循环与无环

标准 LangChain 是**无环的**（顺序式）。
LangGraph 是**有环的**。
- **循环的力量**：智能体可以尝试一个工具，看到错误，然后**循环返回**到“思考”节点再试一次。这是 **ReAct** 模式的基础。

---

## 状态管理

**状态模式**是图的“大脑”。
```python
class GraphState(TypedDict):
    messages: Annotated[list[AnyMessage], add_messages]
    plan: list[str]
    is_secure: bool
```
**细微差别**：将 `Annotated` 与 `add_messages` 结合使用，可以让图对历史进行**追加**而不是覆盖，从而保留完整的推理轨迹。

---

## 持久化与检查点

当前 LangGraph 使用**基于线程的持久化**。
- **概念**：每个会话都有一个 `thread_id`。
- **收益**：如果用户在 2 天后回来，智能体仍记得自己在多步骤工作流中的精确位置。
- **时间旅行**：开发者可以从之前的状态“重新运行”某个特定线程，以调试故障。

## 面试题

### Q: 为什么使用 LangGraph，而不是 OpenAI 的“助手 API”？

**强有力的回答：**
**控制力和可移植性**。助手 API 是一个黑盒：你无法看到精确的 prompts，也无法控制逻辑门。LangGraph 是一个**白盒框架**。我可以使用任何模型（OpenAI、Claude、Llama 3.3），精确控制工具何时被调用，并在步骤之间注入我自己的自定义校验逻辑。更重要的是，LangGraph 是**开源**的，并且可以在本地/本地部署环境运行，这对许多企业安全要求来说至关重要。

### Q: 在一个有 20+ 个节点的图中，你如何处理“状态过载”？

**强有力的回答：**
我们使用**状态收窄**。我们不会把整个全局状态传给每个节点，而是为子图定义专门的子状态。我们还使用 **Trim Runnables** 在消息历史送入 LLM 之前对其进行裁剪，确保在持久化层保留“事实”的同时，不浪费 tokens。
