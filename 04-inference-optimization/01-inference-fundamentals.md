## 推理的两个阶段

LLM 推理不是单一操作；它由两个不同的计算阶段组成。

### 1. 预填充阶段（Prompt 处理）
模型在一次前向传递中处理整个输入提示词。
- **计算**：高并行矩阵乘法。
- **瓶颈**：**计算受限**（受 GPU TFLOPS 限制）。
- **时间复杂度**：$O(N)$，其中 $N$ 为输入长度（但经过并行化）。

### 2. 解码阶段（Token 生成）
模型逐个生成 token，每个 token 都依赖前面的 token。
- **计算**：顺序处理，每次只处理权重矩阵的一行。
- **瓶颈**：**内存受限**（受内存带宽限制）。
- **时间复杂度**：$O(M)$，其中 $M$ 为输出长度（顺序执行）。

## 瓶颈：计算受限 vs. 内存受限

理解系统的瓶颈所在，对于选择合适的优化至关重要。

| 阶段 | 瓶颈 | 原因 | 主要优化 |
|-------|------------|------|----------------------|
| **预填充** | 计算（FLOPs） | 并行处理会让 GPU 的算术单元饱和。 | FlashAttention，FP8/FP16 精度。 |
| **解码** | 内存带宽 | 对于 *每一个 token*，都必须从 VRAM 加载权重。 | 量化（4 位）、GQA、批处理。 |

**内存墙洞察**
随着模型变得越来越大，内存带宽（HBM3/HBM3e）的增长速度并没有像计算能力（TFLOPS）那样快。这使得解码阶段成为生产优化的首要目标。

## 性能指标

| 指标 | 全称 | 目标 | 重要性 |
|--------|-----------|------|------------|
| **TTFT** | Time To First Token | < 200ms | 用户感知响应速度。 |
| **TPOT** | Time Per Output Token | < 30ms | 阅读速度和对话流畅性。 |
| **Throughput** | Tokens/Second (Agg) | 最大化 | 决定每次查询成本。 |
| **Latency** | End-to-End Time | < 2.0s | 智能体完整往返耗时。 |

## 硬件赋能的优化（FP8）

**FP8（8 位浮点数）**是 H100 和 B200 GPU 上推理的原生精度。

- **收益**：比 FP16/BF16 快 2x，且精度损失可忽略（<0.1%）。
- **工作原理**：使用比 Int8 更小的尾数和更大的指数，使其无需复杂校准即可更准确地表示 LLM 激活值的动态范围。

**资深级细节**：服务框架现在使用 **动态 FP8 缩放**，它会按层调整量化尺度，以防止异常值破坏整个模型的逻辑。

## 面试题

### 问：为什么 LLM 生成比分类更慢？

**强答案：**
分类是一个“仅预填充”的任务；它在一次并行传递中处理整个输入并产生单个输出，因此计算效率最优。然而，LLM 生成是**自回归**的。每个 token 都依赖前一个 token，迫使系统进入顺序的“解码”循环。由于这个循环中的每一步都受内存限制（加载千兆字节级权重来产生毫克级数据），系统的大部分时间都消耗在等待内存传输，而不是做数学计算。

### 问：如何优化 TTFT 与 TPOT？

**强答案：**
要优化 **TTFT**，必须优化预填充阶段：使用 FlashAttention-3，提高计算并行度（张量并行），或者使用前缀缓存（Prefix Caching）为常见提示词跳过整个预填充阶段。  
要优化 **TPOT**，必须优化解码阶段的内存带宽：使用量化（4 位权重）减少从 VRAM 移动的数据量，使用分组查询注意力（GQA）减小 KV cache 大小，或者使用推测解码（Speculative Decoding）在一次内存加载中生成多个 token。
