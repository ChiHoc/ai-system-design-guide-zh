# 推测解码

推测解码（Speculative Decoding）是一种如今已成为标准的技术，它允许大模型（LLMs）在一次前向传播中生成多个 token，从而有效打破顺序解码中的内存带宽瓶颈。

## 核心概念

LLM 解码受内存带宽限制：为了生成一个单独的 2 字节 token，却要加载 140GB 权重（70B 模型），这非常低效。  
**推测解码** 使用一种更便宜的方法来“猜测”接下来的 $N$ 个 token，并让大模型在一次并行的“类 Prefill”传递中把它们全部验证。

---

## 草拟-验证范式

1。**草拟**：一个小而快的“草拟模型”（例如 1B 或 7B）生成 $K$ 个候选 token。  
2。**验证**：大的“目标模型”一次处理全部 $K$ 个 token。  
3。**接受**：使用目标模型的 logits 来接受或拒绝候选项。如果 token $i$ 被拒绝，则其后的所有 token 都会被丢弃。

| 模型 | 大小 | 速度 | 每个 token 的延迟 |
|-------|------|-------|-------------------|
| **草拟** | 1B | 快 | 5ms |
| **目标**| 70B| 慢 | 50ms |
| **推测式**| - | **快**| **15ms - 25ms** |

**最终结果**：在**质量零损失**的前提下，实际耗时加速 2 倍到 3 倍。

## Medusa 与多 token 头

业界已经从单独的草拟模型（会增加 VRAM 开销）转向了 **Medusa Heads**。

- **它是什么**：附加在目标模型最后一层上的额外“头”（小型线性层）。
- **工作方式**：不是只预测 token $t+1$，而是 Head 1 预测 $t+1$，Head 2 预测 $t+2$，依此类推。
- **收益**：不需要第二个模型；在 VRAM 只增加很少的情况下可实现 2.5x 加速。

---

## 前瞻解码

另一种方法是利用模型自身过去的隐藏状态来寻找重复模式（n-gram），从而“向前看”并预测未来的 token。
- **最适合**：结构化数据、代码，以及高度重复的技术写作。

---

## 面向硬件的推测

前沿推理框架（vLLM、TensorRT-LLM）现在使用 **动态草拟长度**。
- 如果 GPU 利用率不足（小批量），系统会增加草拟 token 数（$K$）。
- 如果 GPU 已饱和（大批量），系统会减少 $K$，以优先保证吞吐量而非单个请求延迟。

## 面试题

### 问：为什么推测解码在高温度的创意写作场景下效果不好？

**强回答：**  
推测解码依赖于“草拟模型”能够准确预测“目标模型”会说什么。在高温度的创意写作中，概率分布更“平坦”，模型会被鼓励选择更不可能的 token。这会导致**接受率**非常低（草拟模型的猜测经常被拒绝）。当一个猜测被拒绝时，目标模型的并行传递就浪费了计算，而系统会回退到标准的顺序解码，同时还额外带上草拟模型的延迟开销。

### 问：Medusa 与传统推测解码有何不同？

**强回答：**  
传统推测解码需要一个独立、更小的模型（草拟模型），它会占用额外的 VRAM，并且需要自己管理 KV cache。相对地，Medusa 会在基础模型的最终隐藏状态上增加多个“头”。每个头都被训练为预测不同的偏移量（例如，下一个 token、next+1、next+2）。这消除了对第二个模型的需求，并最小化了步骤之间的通信开销，因为所有“猜测”都在一次前向传播中、同一个基础模型架构内生成。
