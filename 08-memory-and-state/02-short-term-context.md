## 上下文生命周期

上下文会经历三个阶段：
1. **接收**：用户查询 + 最近历史 + 系统指令。
2. **处理**：GPU 为新 token 计算 KV 缓存。
3. **驱逐**：在达到上限后移除旧 token，为新 token 腾出空间。

---

## KV Cache 分块

现代推理引擎（vLLM、TensorRT-LLM）使用 **PagedAttention**。
- **思路**：不再为上下文分配一整块连续的 GPU 内存，而是将内存拆分为 **Blocks（块）**（pages，页）。
- **效率**：通过 **60-80%** 降低内存碎片化，从而在同样的硬件上支持显著更大的批大小和更长的上下文窗口。

## 前缀缓存

这对任何生产级 LLM 技术栈来说，都是 **低延迟的圣杯**。
- **问题**：每次智能体调用 LLM 时，都会发送相同的 2,000-token 系统提示词 + 50 个工具模式。这会浪费计算资源。
- **解决方案**：**持久化前缀缓存**。服务器把提示词中“静态”部分（前缀）的 KV 缓存保留在内存中。
- **结果**：你只需要为消息中 *新的* 部分付出（并等待）计算成本。

## 滑动窗口与总结

| 方法 | 机制 | 优点 | 缺点 |
|--------|-----------|-----|-----|
| **滑动窗口** | 精确保留最后 N 个 token。 | 对最近内容保真度高。 | “多利”效应（忘记开头）。 |
| **总结** | 将旧轮次压缩为文本。 | 保留“关键信息”。 | 会丢失细微差别/格式。 |
| **混合** | 保留最后 10 轮 + 1 总结。 | 兼得两者之长。 | 复杂度略高。 |

## 上下文压缩

当前前沿模型支持 **Prompt Hardening（提示词加固）**。
- **选择性丢弃**：自动剥离前几轮中无关的 “Thought” 块，以节省空间。
- **Token Pruning（token 剪枝）**：在发送给 “Reasoning” 模型之前，使用更小的模型把一段很长的用户消息重写成一个 50% 更短、等价的提示词。

## 面试题

### Q: “Model Context Window（模型上下文窗口）” 和 “Application Context Window（应用上下文窗口）” 有什么区别？

**强回答：**
**模型上下文窗口** 是由架构定义的硬性上限（例如 GPT-4o 的 128K）。**应用上下文窗口** 是工程师为了管理 **Latency and Cost（延迟与成本）** 而设置的配置上限（例如 16K 限制）。在生产中，我们很少在每一轮都用满整个模型窗口，因为随着上下文变长，注意力开销也会增加，从而导致生成变慢。我们会使用一个 **Buffer Zone（缓冲区）**，为模型的新回复预留空间。

### Q: “Prefix Caching（前缀缓存）” 会如何改变你设计 System Prompts（系统提示词）的方式？

**强回答：**
它会迫使我把 **静态内容放前面**，把 **动态内容放后面**。早期的 LLM 模式经常把用户姓名或日期放在最顶部，这会破坏前缀缓存。我会把 “Immutable Rules（不可变规则）” 和 “Tool Schemas（工具模式）” 放在开头，把每轮都会变化的 “User Context（用户上下文）” 放在末尾。这样可以确保前 5,000 个 token 在所有用户之间都是相同的，从而最大化推理服务器上的缓存命中率。
