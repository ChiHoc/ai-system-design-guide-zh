# ADR-0001：使用独立中文镜像仓库和本地 Codex 同步

- 状态：已接受
- 日期：2026-07-20

## 背景

中文译文会改写上游绝大多数 Markdown 行。若直接使用 GitHub Fork 的常规合并流，路径虽相同，却会在每次同步时产生大量没有语义价值的逐行冲突。把翻译 API 放入 GitHub Actions 还需要长期保存模型凭据，且难以稳定保护社区已经修正的中文表达。

## 决策

项目使用独立公开仓库。`translation-state.json` 以完整 commit SHA 和 Git blob SHA 记录上游事实基线；本地 Codex 每日获取上游，采用“旧英文、旧中文、新英文”三方更新受影响内容，并在独立上下文完成第二遍校对。校验通过后只创建待审 Pull Request，不自动合并。

## 后果

- 上游结构与中文路径保持一一对应，但 GitHub 不显示 Fork 关系。
- 同步依赖一台可运行 Codex 任务的本地主机；离线期间的更新在下次运行补齐。
- 上游历史非快进、三方定位不可靠、验证失败或主分支竞态都会停止流程，由唯一阻塞 Issue 承载恢复信息。
- 发布只发生在人工合并后的 `main`，仓库和站点不会公开半成品批次。

## 未选择的方案

- GitHub Fork：翻译导致持续的大面积行冲突。
- GitHub Actions 调用翻译 API：需要托管模型凭据，且语义校对与社区修正保护更难控制。
- 保存英文副本：会增加重复内容与漂移面；需要原文时按状态表中的 commit 从上游读取。
